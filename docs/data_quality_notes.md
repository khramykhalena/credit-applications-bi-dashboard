# Data Quality Notes

## 1. Базовая таблица application_train
- `application_train` — это главный слой витрины, где `1 строка = 1 заявка`.
- Дубликатов по `SK_ID_CURR` нет.
- Целевая переменная несбалансирована: доля `TARGET = 1` существенно ниже доли `TARGET = 0`.

## 2. Техническая аномалия DAYS_EMPLOYED = 365243
### Наблюдение
Значение `365243` выглядит как техническая заглушка, а не реальный стаж.

### Решение
- создать поле `days_employed_anom_flag`;
- для расчёта `employment_years` заменять `365243` на `NaN`;
- не интерпретировать это значение как реальный трудовой стаж.

## 3. Высокая доля пропусков в части полей application_train
### Наблюдение
Особенно проблемными выглядят жилищные признаки, где доля пропусков доходит примерно до 66–70%.

### Решение
- не использовать такие поля в первом релизе дашборда;
- по ключевым полям явно показывать долю пропусков;
- при необходимости группировать категориальные пропуски в `Unknown`.

## 4. Таблица bureau — слой 1:N
### Наблюдение
На одного клиента приходится несколько записей внешней кредитной истории.

### Риск
Прямой join с `application_train` вызовет дублирование строк и искажение метрик.

### Решение
Сначала строить агрегирующий слой по `SK_ID_CURR`, и только потом присоединять его к базовой таблице.

## 5. Высокая доля пропусков в отдельных полях bureau
### Наблюдение
Сильные пропуски есть, например, в:
- `AMT_ANNUITY`
- `AMT_CREDIT_MAX_OVERDUE`
- `DAYS_ENDDATE_FACT`
- `AMT_CREDIT_SUM_LIMIT`

### Решение
- в основную витрину выносить только устойчивые агрегаты;
- ограничения по качеству фиксировать в документации;
- спорные поля использовать осторожно и не делать на них основу BI.

## 6. Случаи AMT_CREDIT_SUM_DEBT > AMT_CREDIT_SUM
### Наблюдение
Есть строки, где долг больше суммы кредита.

### Решение
- не удалять такие записи автоматически;
- фиксировать их как ограничение качества источника;
- при анализе интерпретировать осторожно.

## 7. Таблица previous_application — второй слой 1:N
### Наблюдение
У клиента может быть несколько прошлых заявок.

### Решение
Перед объединением с текущей заявкой обязательно агрегировать данные по `SK_ID_CURR`.

## 8. Высокая доля пропусков в части полей previous_application
### Наблюдение
Очень плохо заполнены:
- `RATE_INTEREST_PRIVILEGED`
- `RATE_INTEREST_PRIMARY`
- часть стоимостных и процентных полей

### Решение
- не опираться на эти поля в первой версии дашборда;
- использовать устойчивые агрегаты: число заявок, статусы, суммы, глубину истории.

## 9. Практический вывод
Для BI и системной аналитики сырые таблицы напрямую использовать нельзя. Сначала нужен слой с проверками качества данных и агрегирующая витрина, и уже потом — визуализация в BI.
